= XEN Inventory Changelog =

All notable changes to this project are documented in this file.
Format follows Keep a Changelog (https://keepachangelog.com/en/1.1.0/).
Versions follow Semantic Versioning (https://semver.org/).

------------------------------------------------------------------------------
= 1.7.0 — 2026-05-29 =
------------------------------------------------------------------------------

== Fixed ==
* Calendar day-modal Return button — replaced the bare `window.prompt` + empty-
  notes shortcut with a proper "Return Item" modal that includes:
    - Quantity-returning field (visible only when the borrowed qty > 1).
    - Item-condition drop-down (✅ Usable / ⚠️ Slightly damaged / ❌ Unusable).
    - Return Remarks textarea (mandatory, matching the server-side requirement).
  Previously the calendar return action always sent an empty `notes` value and
  no `item_condition`, causing every return from the calendar to fail with
  "Return remarks are required." / "Please select the item condition."

* Inventory display — borrowed-items borrower list is now displayed as centred,
  card-style pills that wrap to the next row instead of a flat horizontal run
  that clipped or misaligned on overflow. Each pill shows the borrower name,
  contact, and borrow meta (qty × due date) stacked vertically in the centre
  of the card, and the list itself is centred within the item card body.

== Security ==
* Added missing `wp_unslash()` wrappers around all `$_GET` / `$_POST` reads
  that were passing superglobals directly to `sanitize_text_field()` or
  `sanitize_key()` without unslashing first:
    - `class-ajax-handlers.php`: `borrow_item()` POST fields (date_due, notes,
      borrower_full_name, borrower_contact, borrow_tags) and
      `get_calendar_events()` GET fields (start, end).
    - `admin/views/borrow-log.php`: xen_search, xen_status, xen_date_from,
      xen_date_to filter inputs.
    - `admin/views/audit-log.php`: xen_action, xen_date_from, xen_date_to
      filter inputs.

= 1.6.7 — 2026-05-30 =
------------------------------------------------------------------------------

== Fixed ==
* Borrow calendar — returned items now remain visible on their original borrow
  date when navigating to previous calendar months. Previously the SQL overlap
  query excluded events whose return date fell before the visible range start,
  causing historical borrow records to disappear.
* Borrow calendar — event chips and day-modal status indicators are now
  colour-coded by return status:
    - Red   = still borrowed / not yet returned
    - Amber = partially returned (some units back, remainder still out)
    - Green = fully returned
  Previously all events were always shown in red regardless of return status.
* Day-modal status chip now shows "Active", "Partially Returned", or "Returned"
  instead of a binary "Active / Returned" label.
* Return button in the day modal is now correctly hidden for fully-returned
  events (previously relied on date_returned presence; now uses return_status).

= 1.6.6 — 2026-05-30 =
------------------------------------------------------------------------------

== Added ==
* Borrower deletion — Admins can now delete a borrower entity from the
  Borrowers list. Deletion is blocked if the entity still has open (unreturned)
  borrows; a clear error message states how many active borrows remain.
  All borrow records are preserved after deletion; they continue to appear in
  the Borrow Log with a "Deleted" badge next to the borrower name. The
  borrower's detail page shows an "Account Status: Deleted" banner.
* Available-quantity hint in the borrow form — the quantity field on both the
  inline modal and the standalone borrow page now shows "X available" so
  borrowers know the current stock before submitting. The standalone borrow
  page header also shows "Available: X" alongside "Total stock: Y".
* Borrow quantity max attribute now uses the real available quantity (not total
  stock), preventing the form from accepting a number greater than what is
  currently borrowable.

== Fixed ==
* Borrow form error messages — when quantity exceeds available stock the modal
  now displays the server's specific message (e.g. "Only 2 units available")
  instead of the generic "An error occurred" fallback. The quantity field and
  hint text are updated automatically to reflect the actual available amount.

= 1.6.4 — 2026-05-29 =
------------------------------------------------------------------------------

== Fixed ==
* PHP single-quoted strings in `inventory-display.php` and `page-borrow.php`
  contained literal `\u2014` sequences which PHP does not interpret as Unicode
  in single-quoted context — causing `\u2014 Select condition \u2014` and

  `$return_notes` (string) and `$item_condition` (string) instead of `$notes`.

== Added ==

* [Frontend — My Active Borrows] Partial return support. When a borrower
  has more than one unit of an item, a "Qty returning" number input now
  appears on the return row. Returning fewer than the borrowed amount
  reduces the outstanding quantity in-place (the row updates without a

* [Admin — Borrow History Meta Box] "Return" button now prompts for

  log row's outstanding quantity and inserts a new 'returned' row for
  the returned portion.

* [Model] InventoryLog::partial_return() — new static method that handles
  both full returns (delegates to close_log) and partial returns (reduces
  original row qty + inserts returned sub-row).

== Changed ==

* [AJAX — xen_return_item] Accepts a new optional 'qty_returned' field.
  When 0 or absent the entire log row is closed (previous behaviour).
  When a positive integer is supplied the partial_return model method is
  called instead.

== Fixed ==

* [Admin — Borrow Log] "Returned" status filter showed all records instead
  of only returned entries. Root cause: the WHERE clause added a literal
  SQL condition ('l.date_returned IS NOT NULL') with no sprintf placeholders,
  leaving $where_args empty. The if($where_args) guard fell through to the
  unfiltered else branch. Fixed by adding a separate elseif ($where_sql)
  branch that executes the query with the WHERE clause but without prepare
  args.

* [Frontend — Login Shortcode] Pages containing [xen_inventory_login] now
  immediately redirect already-logged-in users to /inventory/ via a
  template_redirect hook (HTTP 302), rather than rendering a static
  "You are already logged in" paragraph. The /inventory/login/ clean-URL
  route already behaved correctly; this fix brings the shortcode page into
  parity.

= 1.5.2 — 2026-05-28 =
------------------------------------------------------------------------------

== Bug Fixes ==

* [Moderate] Item card images cropped to fill the square area.
  The `.xen-item-card__img` rule used `object-fit: cover`, which scales the
  photo to fill the 1:1 container, cropping the top/bottom or sides of
  non-square images. The full product photo was never visible.
  Fix: changed to `object-fit: contain` so the entire image is always shown
  inside the card area, letterboxed against the existing light-gray background.
  Same fix applied to the single-item detail page hero image.

* [Critical] [xen_inventory_display] shortcode page breaks when page slug
  equals "inventory".
  Root cause: the `xen_item` CPT was registered with `has_archive => 'inventory'`,
  which adds a rewrite rule `^inventory/?$` that WordPress inserts with higher
  priority than the matching page-slug rule. Visiting /inventory/ therefore
  resolved to the CPT archive template (not the user's page), bypassing the
  shortcode and leaving all XEN CSS/JS unenqueued.
  Fixes applied:
  1. Disabled the built-in CPT archive (`has_archive => false`). The plugin
     uses the [xen_inventory_display] shortcode for frontend display; a
     separate archive URL was never needed and only caused conflicts.
  2. A one-time rewrite-rules flush is now scheduled automatically on the
     first admin page load after upgrade, so the conflict is resolved without
     requiring the user to manually re-save Permalinks.
  3. The asset-enqueue guard now resolves the queried object with
     `get_queried_object()` (more reliable than `global $post`) so shortcode
     CSS/JS load correctly even when the main query context is ambiguous.

= 1.5.0 — 2026-05-28 =
------------------------------------------------------------------------------

== Bug Fixes ==

* [Critical] Calendar assets not loaded on /inventory/calendar/ clean URL.
  When a visitor accessed the dedicated /inventory/calendar/ route, no WP
  post object exists, so the old early-return guard (`is_a($post, 'WP_Post')`)
  caused the entire enqueue function to exit before the `xen_view` query-var
  check was reached. FullCalendar, calendar.js, and the xenCalendar localised
  object were never enqueued, leaving a blank page.
  Fix: moved `$xen_view = get_query_var('xen_view')` before the `$post` guard;
  all `$post->post_content` references now wrapped in a safe `is_a()` check.

* [Moderate] Active borrow events from previous months invisible in calendar.
  The SQL query used `WHERE date_borrowed BETWEEN start AND end`, which only
  showed events whose borrow DATE falls inside the FullCalendar view window.
  Borrows that started before the current month and had not yet been returned
  were completely absent from the calendar.
  Fix: replaced the BETWEEN clause with overlap logic:
  `date_borrowed <= range_end AND (date_returned >= range_start OR date_returned IS NULL)`.
  Active borrows now correctly span from their borrow date through to today
  (or their due date) across all calendar months.

* [Moderate] Wrong locale format passed to FullCalendar.
  `get_locale()` returns WordPress format (e.g. `en_US`) with an underscore
  separator, but FullCalendar expects BCP 47 format (`en-US`) with a hyphen.
  Fix: locale is now converted with `str_replace('_', '-', get_locale())`.

* [Minor] Calendar event popover showed incorrect "Item" field.
  The event `title` property was the combined `Item — Borrower` string, so
  the "Item" row in the click popover displayed the full concatenated label.
  Fix: `item_title` and `borrower` are now stored as separate `extendedProps`
  on each event; the popover reads `extendedProps.item_title` for the item
  name and `extendedProps.borrower + action` for the borrower row.
  The popover label was also renamed from "Action" to "Borrower" to match.
